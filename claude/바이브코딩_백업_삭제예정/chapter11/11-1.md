# 11. 현재의 한계점

바이브 코딩이 개발의 문턱을 낮추고 생산성을 높이는 것은 분명하지만, 만능은 아닙니다. 이 장에서는 현재 바이브 코딩이 가진 한계를 솔직하게 짚어보고, 앞으로 어떻게 발전할지 전망해보겠습니다. 한계를 알아야 더 현명하게 도구를 활용할 수 있습니다.

# 1. AI 코딩의 기술적 한계

## 2.1 컨텍스트 제약

Claude Code가 아무리 뛰어나도 한 번에 처리할 수 있는 코드의 양에는 한계가 있습니다. 보통 10,000줄이 넘는 대규모 프로젝트에서는 전체 구조를 파악하기 어렵습니다. 파일 하나씩 수정할 수는 있지만, 프로젝트 전체의 아키텍처를 고려한 리팩토링 또는 코드 수정이 불가능합니다.

또한 필요 이상의 파일을 읽거나, 함수간의 관계를 모두 고려하지 않습니다. 예를 들어, 여러분이 HTML, CSS, JavaScript의 단순한 코드를 수정하더라도 CSS를 계속해서 덮어써서 코드를 수정하는 일이 발생할 수 있습니다. 컨벤션에 어긋나는 코드를 작성할 수 도 있죠. 규모가 크고 데이터베이스가 복잡하다면 문제는 더 심각할 수 있습니다. 새로운 테이블이 중첩해서 생기고, 나중에는 수정이 불가능한 지경에 도달할 수도 있습니다.

AI가 개발해야 할 것을 더 세분화 하고, 요구사항을 템플릿화 하여 일관성 있는 주문을 사내에 내제화 하는 것이 바이브 코딩을 통한 프로덕트 생산성을 올리는 관건이라 할 수 있겠습니다. 그렇지 않으면, 유지보수하는데 더 많은 시간을 보내야 할 수도 있습니다.

## 2.2 최신 정보의 부재

AI 모델은 특정 시점까지의 데이터로 학습되어 있습니다. 새로 나온 라이브러리나 프레임워크의 최신 버전, 며칠 전에 발견된 보안 취약점 같은 정보는 알지 못합니다. "React의 새로운 기능 OOO을 활용해서 만들어줘"라고 요청해도, 학습 데이터에 없다면 제대로 구현할 수 없습니다. 또한 그 내용을 알고 있어도 참고할만한 레퍼런스가 많지 않을 경우에는 이전 버전의 코드를 작성하는 경우가 많습니다.

위니브에서는 특정 라이브러리를 업데이트 하는 사례가 있었습니다. 업데이트가 된지 꽤 된 라이브러인데, ChatGPT나 Claude에게 아무리 질문을 해도 제대로 동작하는 코드를 작성하지 못했습니다. 이유는 레퍼런스에 있었습니다. 사용하는 사람이 적으니, 해당 코드를 제대로 생성을 못하더군요.

특히 빠르게 변화하는 웹 개발 생태계에서는 이것이 큰 문제가 됩니다. 개발자는 항상 생성된 코드가 최신 베스트 프랙티스를 따르는지 검증해야 합니다.

# 3. 보안과 프라이버시 이슈

## 3.1 민감한 정보 노출 위험

회사의 코드나 고객 데이터를 AI에게 보여주는 것은 심각한 보안 위험이 될 수 있습니다. 특히 금융, 의료, 정부 프로젝트에서는 코드 자체가 기밀일 수 있습니다. API 키, 비밀번호, 개인정보가 포함된 코드를 실수로 AI에게 전송하면 큰 문제가 발생할 수 있습니다. 해당 AI가 그 데이터를 학습하지 않는다고 하더라도, 데이터 전송 만으로도 보안 위반이 될 수 있습니다.

## 3.2 생성된 코드의 보안 취약점

바이브 코딩으로 SW를 처음 접하고 만든 사람일 경우 SW를 배우고 SW를 만든 것이 아니기 때문에 보안 취약점에 대한 인식이 전혀 없을 수 있습니다. AI는 "작동하는" 코드를 만들지만, "안전한" 코드를 보장하지는 않습니다.

예를 들어, 처음으로 SW를 개발해봤다면 admin 서비스가 밖에서 검색이 안되어야 한다는 사실이나 /admin 으로 접속이 되면 안된다는 사실 조차 알지 못할 가능성이 있습니다. 패스워드가 3회 틀렸을 시에 재시도 시간 텀을 길게 둔다던지, 비밀번호를 특수문자나 숫자가 함께 포함되게 해야한다던지 하는 기본적인 사항들이 지켜지지 않을 가능성이 있습니다. 또한 비밀번호를 "README.md" 파일에 써서 파일로 공개할 수도 있습니다.

## 3.3 법적 책임의 모호함

AI가 생성한 코드로 인해 문제가 발생했을 때, 책임 소재가 불명확합니다. 저작권 침해, 특허 침해, 오픈소스 라이선스 위반 등의 문제가 발생할 수 있습니다. AI가 학습 데이터에서 본 코드를 그대로 재현했다면, 원저작자의 권리를 침해할 가능성도 있습니다.

실제로 GitHub Copilot이 라이센스 코드를 그대로 생성하는 사례가 발견되어 논란이 된 적이 있습니다. 기업에서 AI 생성 코드를 사용할 때는 법무팀의 검토가 필요할 수 있습니다.

# 4. 의존성과 학습 곡선

## 4.1 AI에 대한 과도한 의존

바이브 코딩에 익숙해지면 기본기가 약해질 위험이 있습니다. 항상 AI에게 물어보고 코드를 생성받다 보면, 스스로 문제를 해결하는 능력이 퇴화할 수 있습니다. 마치 계산기에 의존하다가 암산 능력을 잃는 것과 비슷합니다.

특히 초급 개발자들이 이런 함정에 빠지기 쉽습니다. 왜 그런 코드가 생성되었는지 이해하지 못한 채, 복사-붙여넣기만 반복하면 실력이 늘지 않습니다. 디버깅 능력, 문제 분석 능력, 알고리즘 사고력 등 핵심 역량이 개발되지 않을 수 있습니다.

이렇게 되었을 때 문제는 "문제가 발생했을 때" 입니다. 특히 상용 서비스가 라이브 되고 있는 상황에서 문제가 발생이 되면 문제 해결이 거의 불가능할 수 있습니다.

## 4.2 디버깅의 어려움

AI가 생성한 코드에서 버그가 발생하면, 디버깅이 더 어려울 수 있습니다. 자신이 작성하지 않은 코드이므로 로직을 완전히 이해하기까지 시간이 걸립니다. 특히 AI가 너무 "똑똑한" 해결책을 제시했을 때, 그 복잡한 로직을 따라가기 어려울 수 있습니다.

또한 AI에게 "이 버그를 고쳐줘"라고 요청해도, AI가 문제의 원인을 정확히 파악하지 못할 수 있습니다. 결국 개발자가 직접 콘솔을 확인하고, 로그를 분석하고, 단계별로 디버깅해야 합니다.

# 5. 비용과 접근성

## 5.1 유료 서비스의 비용 부담

Claude Pro, GitHub Copilot, ChatGPT Plus 등 좋은 AI 코딩 도구들은 대부분 유료입니다. 개인 개발자에게는 월 20~30달러가 부담이 될 수 있고, 기업에서 팀 전체가 사용하려면 상당한 비용이 발생합니다.

무료 버전은 사용량 제한이 있거나 성능이 떨어집니다. 복잡한 프로젝트를 진행하다 보면 금세 한계에 도달합니다. 결국 제대로 활용하려면 비용을 지불해야 하는데, 이것이 진입 장벽이 될 수 있습니다.

위니브에서는 이런 장벽을 없애기 위해 항상 최신의, 최고의 모델을 쓸 수 있도록 합니다. 개인당 수십만원이 들어도 회사가 지불해줍니다.

이러한 진입장벽이 이제는 회사의 경쟁력을 좌우하는 시기가 되었습니다.

## 5.2 인터넷 연결 의존성

대부분의 AI 코딩 도구는 클라우드 기반이므로 안정적인 인터넷 연결이 필수입니다. 인터넷이 불안정한 환경에서는 사용하기 어렵고, 오프라인에서는 전혀 사용할 수 없습니다. 비행기나 지하철에서 코딩할 때는 AI의 도움을 받을 수 없습니다. 특히 회사가 인트라넷을 사용하면 사용할 수 없습니다.

또한 서비스 장애가 발생하면 작업이 중단됩니다. OpenAI나 Anthropic 서버에 문제가 생기면, 전 세계 개발자들이 동시에 영향을 받습니다. 이런 의존성은 프로덕션 환경에서 위험 요소가 될 수 있습니다.

# 6. 그럼에도 불구하고

이렇게 많은 한계가 있음에도 불구하고, 바이브 코딩은 계속 발전할 것입니다. 현재의 한계들은 시간이 지나면서 하나씩 해결될 것입니다. 컨텍스트 윈도우는 계속 확장되고 있고, 보안 기능은 강화되고 있으며, 비용은 점점 낮아지고 있습니다.

중요한 것은 AI를 도구로 인식하고 현명하게 활용하는 것입니다. AI가 모든 것을 해결해주는 마법이 아니라, 개발자의 생산성을 높이는 도구라는 것을 이해해야 합니다. 한계를 인지하고, 적절한 곳에 적절한 방식으로 사용한다면, 바이브 코딩은 강력한 무기가 될 것입니다.

회사라면 회사에서는 AI 도구와 인간이 어떻게 역할 분담을 하여 업무를 진행할지 체계적으로 정리를 해두어야 합니다. 회사는 이것을 개인에게 맡겨두어서는 안됩니다. 회사 차원에서 정리를 해야 합니다. 예를 들어, 사내에 있는 디자인 시스템을 프롬프트로 옮기고, 모두가 이 프롬프트를 사용해서 페이지를 만들 수 있도록 전략적으로 접근해야 합니다.

미래에는 AI가 코드를 생성하는 것을 넘어서, 시스템을 설계하고, 아키텍처를 제안하고, 자동으로 테스트하고 배포하는 수준까지 발전할 것입니다. 그때까지 우리는 현재의 도구를 최대한 활용하면서, 동시에 기본기를 탄탄히 다져야 합니다. 바이브 코딩의 시대는 이제 막 시작되었을 뿐입니다.

시장은 이미 변화의 조짐이 보입니다. 아는 대표님은 프론트엔드부터 백엔드, 디자인까지 50만원에 개발을 하는 자동화 AI 개발 시스템을 구축하여 서비스하고 있습니다. 이전 같았으면 말도 안되는 가격입니다. 1 ~ 3 페이지 템플릿으로 찍혀져 나오는 개발이 아니라면요. 팀장, 프론트엔드 개발자, 백엔드 개발자, 디자이너 적어도 이렇게 4명은 필요하니까요.

변화는 시작되었습니다. 변화를 주도해가셨으면 좋겠습니다.
